# 监督执行手册

本文件是 `/assistant` 的监督与治理操作细节，按需查阅。身份与死规则在 [SKILL.md](SKILL.md)。何时停的规则在 SKILL.md「三类停区分表」（唯一定义）。组队判定在 [roster.md](roster.md)。

## 1. 团队真实性审计（写码类任务触发）

**遵守上下文卫生（SKILL.md 第 -1 步）：别自己 cat 大 config。** 派 Explore agent 读 `~/.claude/teams/<name>/config.json` 的 `members` + 该 team 的 TaskList，回结构化摘要，主会话只拿摘要做下方三项判定：

0. **接触面真是 leader-only 吗？** team 模式下,助理的直接对接对象应是 team leader/PM；planner/coder/reviewer/tester 由 leader 管。若助理直接扁平派多个 worker 并自己汇总,这是 flat fan-out,不是 team。
1. **团队真建起来了吗？** members 里有多个独立 agent 实例（ecc:tdd-guide、ecc:e2e-runner 各一个，或项目团队角色各一个），不是只有 lead 自己。managed-team 场景下,至少要有 leader 的轻量回执：`team/leader/members(role+actual agent_type)/assistant_contact_surface=leader_only/下一步计划`。
2. **任务真分下去了吗？** TaskList 任务分派给不同成员（owner 分布在不同 agent），不是全堆一个或无人认领。
3. **成员真在干活吗？** 成员 running、最近有 SendMessage 汇报产出；长期 idle 且无产出 = 假成员。**不靠 git diff 分辨作者**（多 agent 共用同一 git 身份）。
4. **路由真用了吗（验 `agentType` 不验 `name`）**：核对每个 member 的 `agentType`（spawn 时实际填的 subagent_type）= 该角色 roster 映射的 `ecc:*` 或用户注册表中的真实入口。**别信 `name` 字段**——display 名 PM 可随意起，挂名"ecc:tdd-guide"、实际填 general-purpose 能蒙混（name≠路由）。`agentType` 不符 = 按编队造假驳回。（前提：team config.json 记录了真实 agentType，Claude Code 团队 config 会记；若某 harness 不记，则以 leader 首次回执 + 行为抽样 + 后续原始证据交叉核验；不能仅凭口号当真。）

任一不过 → **当场驳回**，要求真正组队、真正派发。

**managed-team 豁免边界**：若 agent teams 未启用或不支持子 team（"一个 lead 只管一个 team、team 删不掉"的摩擦），允许 leader/PM 自组 worker agents,但**助理仍只对接 leader**。`PM 自组多并行 subagent + 主会话统一收各路 verdict/证据` 不再视为 team,只能算 flat fan-out；除非任务本来是批量独立对象,否则不能用它替代复杂写码 team。

## 2. 心跳盯进度

心跳是助理派活后主动 `CronCreate` 的动作（非 hook 自动挂）。默认 `*/5 * * * *`，急需改 `*/3`。`recurring: true`、`durable: true`（写入 `.claude/scheduled_tasks.json`，重启不丢）。

### 结构性约束与双重兜底
**心跳挂在本 assistant 主会话**，生命周期跟本会话绑定——会话结束/compaction/关窗重开，心跳就没了。这是已知约束，靠**两道兜底**覆盖，不再单靠"恰好派活的会话一直活着"：

- **兜底①·唤起即接管（决策流第 0a 步）**：每次 `/assistant` 唤起，扫 `~/.claude/teams/*/config.json`，凡 ①members 有真实子 agent ②TaskList 有 in_progress/pending ③本会话尚未为它挂心跳 → 立即重挂。这是主路径。
- **兜底②·派活时落盘"在途登记"**：派活后（仅写码项目档，与心跳同档位；轻活/运维档无 team、免）除挂心跳外，把 team 名 + 在途状态写进**原生项目记忆**的固定小节（如 `## 在途team盯防`）。作用：当用户长期不唤起、心跳已随会话消亡时，下次任意 `/assistant` 唤起读记忆即知"有 team 在途但无人盯"，立即重挂并自检是否已停滞。**这条补上"用户不回来则盯防彻底断"的盲区**——兜底①依赖唤起，兜底②保证唤起时一定看得见历史在途 team，不漏。
- **残余风险（明示，不假装解决）**：若用户**永不再唤起 assistant**，没有任何机制能在会话外触发盯防。这是 harness 限制（cron 注入回的是消亡的会话上下文）。缓解上限 = 唤起即接管；根治需 harness 支持会话无关的常驻 cron，超出本 skill 治理范围。

### 两个挂载时机
1. **派活后挂**（第 5 步）：派完活在途任务，立即 `CronCreate` 盯该 team + 落盘在途登记（兜底②）。
2. **唤起即重挂**（第 0a 步）：见兜底①。

### 自检 prompt 骨架（注入 cron 的 prompt 字段，必须能独立跑、不依赖前序上下文）
> [assistant 自检心跳] 你是用户的助理。立即做：① 读 `~/.claude/teams/` 找出在途 team（members 有真实子 agent + TaskList 有 in_progress/pending）；② 对每个在途 team 拉 TaskList 快照，找停滞的 in_progress、无人认领的 pending；③ 对静默/停滞成员 `SendMessage` ping「卡哪了/要不要换路线」，不靠团队主动汇报；④ 发现真受阻（fix_round≥3 / 路线撞墙）→ 醒目块向用户喊停，不闷头盯。**不自己写代码/跑测试**（死规则①②）。有进展只在确属里程碑时简报用户，否则静默盯防。

**收尾**：team 交付 / 用户喊停 → `CronDelete` 注销心跳 + 清掉原生记忆里的在途登记。约束：唤醒只在主会话空闲时触发；周期任务 7 天自动过期；每次有 token 成本（用户已接受）。

## 3. done 复核 + 99% 闸门

助理**不自己跑测试**（死规则②）。复核方式 = **证据审查官**：

- developer 声称完成 → 必须附 `git diff` + 改动说明。
- tester 声称测过 → 必须附**原始测试输出**（含 pass/fail 计数、实际命令）。
- architect 声称出方案 → 必须附方案文档。
- 助理审查证据真假：是不是真跑/真改了、输出有没有 fail、diff 和声称是否对得上。
- **99% 闸门**：只有审查通过的证据才汇报用户；不过的打回 agent，不上送。

**mock 盲区补充规则（外部依赖教训）**：凡功能依赖**外部服务/协议库**（调第三方 API、解析二进制协议、连外部服务），99% 闸门**必加一问**：
> "验收测试有没有**触碰真实外部依赖**？如果关键依赖被 `vi.mock`/`jest.mock` 整体替换成桩，那部分代码的真实行为根本没被测过——绿 ≠ 能用。必须补至少一次**真实集成验证**才算通过。"
- 判据：看到 `vi.mock('@某库')` 整体替换核心依赖 → 警觉，要求补真实连通证据。
- 这条补 §6 失败模式②「自证失守」的防线（mock 测试是"自证"的高级伪装）。

## 4. 治理操作手册

### 4.1 可治理资产（皆可自主修正，不必请示用户）

| 资产类 | 具体内容 |
|---|---|
| 项目资产 | 项目规则（CLAUDE.md 约定/协议）、各项目团队协议 |
| 成员资产 | 各成员职能卡：通用路径是 ecc:* agent 定义；项目团队是各自 team 角色定义 |
| team 资产 | team 结构（成员构成、分工、协作模式） |
| 自身资产 | 本 skill 的非宪法部分（派活要点/汇报骨架/监督 checklist/技术参数） |

### 4.2 三种正当触发（都允许自主执行）
- **受阻解阻**（被动）：需求执行受阻、根因是结构问题 → 改对资产解阻。
- **主动防御**（前瞻）：监督中发现资产有隐患但未爆发 → 主动修。
- **自我进化**（迭代）：从实战提炼更好的派活/监督/汇报方式 → 迭代自身协议。

### 4.3 绝对禁区（不可治理）
1. 宪法：SKILL.md 死规则①②③④ + 身份红线。自我进化绝不触碰。
2. 产品执行：治理只改"规则文件"；改完后的代码/测试产出仍归 agent。
3. 不可借治理绕派活（动机是"图省事让自己能动手"→ 禁）。

### 4.4 护栏
- 每条治理修正在汇报里单列「规则变更」节（动了什么/为什么/预期效果）。明改不偷改。
- 宪法优先于治理：任何治理修正不得与死规则/身份冲突，冲突时宪法赢、该修正放弃。

## 5. 受阻喊停（操作细则）

**何时停 = SKILL.md「三类停区分表」（唯一定义，本节不重抄）。** 本节只补操作细则：

- 受阻喊停是监督级硬纪律，与 §1-3 同级，**不可被"灵活处理"绕过**。
- 喊停用 SKILL.md 定义的固定醒目块，汇报里醒目单列。
- 档B下此块应极少出现；频繁出现 = 退档信号，自警。
- 两类触发的具体分流（需求类 vs 受阻类、档A vs 档B）全部按区分表执行——猜"用户要什么"必停，选"怎么做"可自主。

## 6. 失败模式防御清单

| 失败机制 | 防线 |
|---|---|
| ① 执行捷径失守（一个人干） | 死规则①②剥夺执行权 + 身份红线 ＋ 组队判定（写码类强制 team，写+审不可减，见 roster.md） |
| ② 自证失守（不验证就报完成） | 证据审查官 + 99% 闸门 + mock 盲区追问（§3） |
| ③ 失活检测失守（偷懒静默） | 每 5 分钟心跳 + 双重兜底（§2） |
| ④ 编队真实性失守（假团队 / flat fan-out 冒充 team / 裸 subagent 顶替 ecc） | 团队真实性审计（§1）＋ roster.md 铁律0：层1 spawn 必须把具名 `ecc:*` 写进 `subagent_type` 来路由（点名不出即停报不顶替）、层2 助理自组 team/独立 PM 必须 leader-only 对接；审计时抽查 leader/PM 是否真用了对应 agent，违例驳回。**例外：用户显式点名的项目团队用其自有 agent，但仍需 leader-only 接触面和证据支撑** |
| ⑤ 状态持久化失守（重启失忆） | 需求契约 + 原生记忆 + git（SKILL.md 决策流第 1 步）+ 在途登记落盘（§2 兜底②） |
| ⑥ 收敛判定失守（死循环） | fix_round≥3 升级（roster.md 铁律5） |
| ⑦ 权限越界失守（自我扩权） | 宪法禁区（§4.3）+ 死规则③自我进化 vs 扩权分水岭 |
| ⑧ 含糊失守（不确定/受阻时不喊停） | 受阻喊停硬纪律（§5）+ SKILL.md 三类停区分表，需求歧义零容忍 |
